# 414th Air Defense Planning Notes

## BARCAP vs SCRAMBLE Doctrine

In this note, "SCRAMBLE" is doctrinal shorthand for reactive interceptor behavior; the
live code path is the upstream PR `#782` QRA reserve plus Moose dispatcher system.

Goal for a future planning rework:

- BARCAP should be forward deployed to keep enemy aircraft out of friendly airspace.
- SCRAMBLE should be a reactive border-defense/QRA mission that launches or commits only when enemy aircraft cross into friendly territory or a defined border zone.

Current implementation notes:

- BARCAP racetracks are generated by `CapBuilder.cap_racetrack_for_objective(..., barcap=True)`. For a friendly control point target, this already places the track forward from the friendly CP toward the nearest enemy airfield while avoiding enemy threat zones.
- QRA now follows upstream PR `#782`: BARCAP-capable squadrons hold aircraft back via per-squadron `intercept_reserve`, mission generation emits late-activated parked intercept templates plus `intercept_entries`, and `resources/plugins/intercept/intercept-config.lua` uses Moose `AI_A2A_DISPATCHER` to spawn the live interceptors in air.
- Fielded QRA count is per squadron, not per airfield, and is clamped by owned aircraft and available untasked pilots (`qra_resource_count(...)`).
- This means current QRA behavior does not depend on untasked OPFOR filler aircraft being spawned at airfields; dedicated intercept templates are emitted separately before untasked filler aircraft.

Desired rework:

- Separate BARCAP targets from SCRAMBLE targets instead of feeding both from `vulnerable_control_points()`.
- Define a forward defensive CAP line from active front lines, threat boundaries, or friendly territory edges rather than only CP vulnerability.
- Keep the per-squadron reserve model, but continue tuning how far forward BARCAP sits versus how aggressively QRA bases commit.

Existing balance levers to check first:

- Retribution already exposes campaign doctrine tuning under Settings -> Campaign Doctrine.
- Before adding new planner settings or hard-coded distances, test whether existing doctrine values can produce the desired BARCAP/SCRAMBLE behavior.
- Exposed settings that matter today:
  - `Desired BARCAP on-station time` (`desired_barcap_mission_duration`) controls BARCAP duration and indirectly controls how many BARCAP rounds are planned across the mission window.
  - `Airbase threat range (NM)` (`airbase_threat_range`) controls whether a friendly control point is considered vulnerable. That still matters for planned defensive air missions such as BARCAP, but QRA sizing now comes from per-squadron `intercept_reserve` values rather than a package planner task.
  - `TARCAP threat buffer distance (NM)` (`tarcap_threat_buffer_min_distance`) affects TARCAP racetrack placement, not BARCAP. It is useful context but not the lever for forward defensive BARCAPs.
- Faction doctrine constants, not currently editable in Settings, also shape BARCAP behavior:
  - `cap_min_distance_from_cp` / `cap_max_distance_from_cp` place the BARCAP racetrack forward from the defended control point toward the nearest enemy airfield.
  - `cap_min_track_length` / `cap_max_track_length` determine racetrack length.
  - `cap_engagement_range` determines the commit/engagement bubble and is also used to keep BARCAP commit range outside enemy threat zones.
- Current defaults: modern BARCAP uses 10-40 NM from the defended CP, 15-40 NM racetracks, and 50 NM engagement range; coldwar uses 8-25 NM, 12-24 NM racetracks, and 35 NM engagement range; WWII uses 0-5 NM, 8-18 NM racetracks, and 20 NM engagement range.

Implementation areas to revisit:

## Planning rework - historical branch notes

This section is historical context only. The branch described below predated the later
cleanup that switched Retribution onto upstream PR `#782`'s per-squadron QRA reserve and
Moose dispatcher path.

- **CAP timing & overlap.** `MissionScheduler` now schedules land-CP BARCAP as
  overlapping waves (offset `patrol_duration - barcap_overlap_time`) with a
  jittered first wave, instead of front-loaded back-to-back waves all arriving at
  t=0. `TheaterState.from_game` plans enough rounds for the overlap. New
  `barcap_overlap_time` Campaign Doctrine setting (default 15 min; 0 = legacy).
  Carrier/AEWC scheduling unchanged.
- **Forward CAP line.** `ObjectiveFinder.vulnerable_control_points()` now also
  defends any friendly CP that anchors an active front line (`has_active_frontline`),
  so coverage reaches raids inbound to rear income points - not only bases near an
  enemy airfield. OPFOR offensive-roll gate still respected.
- **QRA path superseded.** The branch's border-triggered ramp-scramble system was later
  replaced by upstream PR `#782`. The live implementation now uses per-squadron
  `intercept_reserve`, parked late-activated intercept templates,
  `dcsRetribution.Intercept` Lua data, Moose `AI_A2A_DISPATCHER`, and
  survivor-based `intercept_survivors` mission-end accounting.
- **Doctrine setting fixes (Workstream E).**
  - Fixed the OPFOR/aggressiveness direction inconsistency. The setting is the
    *ratio of threat OPFOR ignores* (0 = defend everything, 100 = fully
    offensive), per the label and `PackagePlanningTask._get_weighted_threat_range`.
    `ObjectiveFinder.vulnerable_control_points` had it inverted (`randint >
    aggressiveness`); now `randint <= aggressiveness`, so both call sites agree.
    **Behavioural note:** at the default value 20, RED now plans offensively for
    only ~20% of its CPs (defends ~80%) instead of the old ~80% offensive. This
    is intended - it makes RED actually defend its airspace/income points - but
    if RED feels too passive offensively, raise `opfor_autoplanner_aggressiveness`
    (the knob is now coherent, so higher = more offensive everywhere).
  - Bumped two engagement-range defaults so AI A2G is less passive:
    `cas_engagement_range_distance` 10 -> 15 NM, `armed_recon_engagement_range_distance`
    5 -> 10 NM.
  - Verified-but-unchanged: `max_mission_range_planes` is a *floor*
    (`max(aircraft.max_mission_range, setting)`), not a cap, so raising it risks
    tasking short-legged jets beyond fuel; left at 150. `airbase_threat_range`
    left at 100 (forward CAP no longer depends solely on it). `player_startup_time`
    left at 10 (player QoL, not an OPFOR lever).

## Mission-preference (`tasks:`) value rubric

The auto-planner ranks aircraft for a task by these YAML weights (higher = chosen
more often: `AircraftType.priority_list_for_task`). Rubric for editing them:

- A value expresses **how good this platform is for this task relative to its
  contemporaries in its era**, not raw desire to fly it.
- For a platform whose *role* is air superiority/interception (e.g. M-2000C,
  F-15C, MiG-31, Gripen-C), its best A2A value should exceed its A2G values - even
  if the airframe can technically drop bombs - or the planner benches it.
- Dedicated maritime-strike bombers (Tu-22M3, Tu-16, H-6J, Tu-95K) should carry a
  high Anti-ship value; it is their signature role. (Anti-ship is scenario- and
  era-dependent - a modern Burke screen may make it suicidal - but the *platform*
  rating should reflect capability.)
- Avoid copy-paste: several blocks are byte-identical across unrelated aircraft.

Tooling:

- `tmp/audit_mission_preferences.py` (read-only) reports per-task distributions,
  A2A platforms benched behind their own A2G, anti-ship outliers, and copy-paste
  clusters. Run it before/after edits.
- `tmp/rebalance_mission_preferences.py` generates a proposal CSV only; it does
  **not** write YAML. The current proposal uses a 60% model / 40% old-value blend,
  preserves low old values as off-role signals unless the archetype makes that
  task central, and applies a small manual archetype override table for bad role
  metadata (`EA-18G`, `CH_JAS39C`, `M-2000C`, `MiG-25RBT`, `VSN_F105G`, and key
  Soviet maritime missile carriers).

Current generated proposal (`tmp/mission_pref_proposal.csv`) improves the
archetype-aware audit from 13 -> 4 A2A-primary platforms benched behind their own
A2G and 12 -> 8 maritime/bomber anti-ship outliers, without applying a blind
256-aircraft YAML rewrite. Keep using the CSV as a review aid, then apply
high-confidence aircraft blocks deliberately.

Applied so far (the two examples the doctrine discussion named explicitly):
`Tu-22M3` Anti-ship 170 -> 800; `M-2000C` A2A tasks 230 -> 460. The audit still
lists ~80 A2A-benched and many anti-ship outliers - these are intentionally left
for case-by-case, in-game-tested tuning rather than a blind mass rewrite, since
the values ship to all Retribution campaigns.
